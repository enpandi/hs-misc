UIL-style HackerRank contest coding stuff (Java/Python)

* there are MANY ways to do the same thing, i am only showing a few.

* more frequent topics (based on the 2018 UIL season) are addressed in more detail, and will generally appear closer to the top of the page (i.e. less scrolling).

* this is more to expose topics than to teach them. however, there are links so that you have somewhere to start and teach yourself :)

* i've omitted C++ because not many of you plan to use it. also i don't think it's suited for this type of programming contest.

 

general tips: read the problem statements and think before you code. not reading the problem statements would often cost my team 5 points back when incorrect submissions were penalized (there should be no incorrect-penalties at the upcoming 7L contest). don't waste time coding solutions that (a) don't address the problem or (b) you don't understand. also, you can often save a lot of coding time just by thinking about the problem a little more.

input

for HackerRank use standard input, NOT file input.

most programs will have you solve the same type of problem multiple times (test cases). there are two main input formats: format 1, which begins with some integer indicating how many test cases follow, and format 2 (slightly more common), which does not.

format 1 example from 2018/S/12 Romero:

2
9/8/2017 3250 14250 5.00 3.00
9/22/2017 2980 15027.50 6.00 2.50

format 2 example from 2018/S/09 Milo:

IV
V
VII
XIX
XX
CV
MDCLXXVIII

different input types have different overall program structures.

format 1 templates:

int T = in.nextInt(); in.nextLine();
while (T-->0) {
    // code
}
xxxxxxxxxx
for _ in range(int(input())):
    # use input() to get more lines

for Java, the in.nextLine() moves the Scanner cursor to the next line, which is very important and has often caused unexpected bugs when left out (we wasted at least twenty minutes during 2019 UIL because of this). alternatively, use parseInt(in.nextLine()).

formatting

primarily addresses alignment and decimals, but it can also be used to clean up long concatenation chains.

here's a relatively simple example from 2018/D/02 Bayani:

input/output:

6.99
12.87
5.44
99.99
115.87
        $   6.99
        $  12.87
        $   5.44
        $  99.99
        $ 115.87
Total = $ 241.16

code:

double total = 0.0;
while (in.hasNextDouble()) {
    double bill = in.nextDouble();
    out.printf("%8s$%7.2f%n", "", bill);
    total += bill;
}
out.printf("Total = $%7.2f%n", total);
total = 0.0
for line in __import__('sys').stdin:
    bill = float(line)
    print(f"{' '*8}${bill:7.2f}")
    total += bill
print(f"Total = ${total:7.2f}")

again, this is just a very simple example, there's a lot more to formatting than what's shown in Bayani.

math

sometimes you need to convert bases. here are some library functions for a few common bases: toBinaryString, toOctalString, toHexString, bin, oct, hex

* almost all math problems can be solved with algebra 1. a lot of problems just give you a formula and tell you to put it in code. don't worry too much about the rest of this section.

probably save these documentation pages just in case: Java, Python

Java-specific:

  • most of the time, you can safely use int.
  • if you need numbers outside of the range then use long.
  • if you need numbers outside of the range then use BigInteger.
  • if you need arbitrarily-high-precision real numbers, use BigDecimal.

Python-specific:

  • the fractions module is pretty good for precision. convert to float when you need to print. i've never tried it, but there's also a decimal module.

avoid using == with double or float

math is faster than looping. for example, x==y but y is significantly faster:

int x = 0; for (int i=1; i<=n; ++i) x+=i;
int y = n*(n+1)/2;
x = sum(range(1, n+1))
y = n*(n+1)//2

this will matter only in rare cases. (Seven Lakes made a problem like this last year (it also required BigIntegers, which suck).)

breadth-first search for shortest paths in mazes and unweighted graphs

Floyd-Warshall: a relatively easy shortest-path algorithm for weighted graphs (fast enough for up to ~500 vertices)

Dijkstra's: a much-less-easy shortest-paths algorithm that's usually much faster than Floyd-Warshall (use if the graph has a lot of vertices)
